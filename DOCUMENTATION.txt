================================================================================
  ACTUAL DATA TOKEN CHECKER — USER GUIDE
================================================================================
  Live site:   https://actual-token-checker.onrender.com
  Local site:  python run_web.py  →  http://127.0.0.1:8080/
  Desktop app: python desktop_app.py

  This guide explains what each part of the app does, in plain language —
  including Quick mode, social scores, news, wallets, Rugcheck / creators,
  the About tab, the Ruggers sell-out tracker, the RugWatch Analyze
  checkbox, yellow Ruggers Upload to cloud, and how flags use local + cloud.

  Everything here is a HEURISTIC from public data. Not financial advice.

--------------------------------------------------------------------------------
1. WHAT THIS APP IS
--------------------------------------------------------------------------------

A research tool for crypto tokens (especially Solana, plus other chains like
Ethereum, Base, and Robinhood Chain). You paste a contract or ticker; the app
pulls public market and risk data and shows it in tabs:

  Overview · Holders · Bundles · Alerts · Maps · About · Ruggers · Logs

It does NOT trade for you, custody funds, or guarantee a token is safe.


--------------------------------------------------------------------------------
2. HOW TO USE THE SITE
--------------------------------------------------------------------------------

  1. Open the site (wait ~1 minute if Free hosting is waking up).
  2. Paste a mint address, 0x address, or name/ticker.
  3. Optional: pick a chain (auto / solana / robinhood / ethereum / …).
  4. Optional: check Quick (faster, less detail — see §3).
  5. Optional: check RugWatch (yellow) — include flagged wallets on Analyze
     (default ON). Uncheck to ignore local + cloud RugWatch lists for this run.
  6. Click Analyze.
  7. Read the tabs. Click blue links to open explorers / X / news.


Navigation bar (compact)
  [logo]  [RugWatch]  |  Docs | GitHub | Stats API | Badge

  Logo stays far left.
  RugWatch (yellow letters) sits immediately to the right of the logo and
    opens the RugWatch website (default http://127.0.0.1:8790/ — set
    rugwatchUrl in web/config.js when you have a public URL).
  Docs → /docs.html on this website (loads web/documentation.txt)
  Also: /documentation.txt (raw text) · /docs (same as docs.html)
  GitHub → https://github.com/Greenwolf30/Actual-Token-Checker
  Footer “Documentation” link opens the on-site guide (not only GitHub).
  Analyze is the search button under the title.

Analyze options (next to the search form)
  quick      — faster market-only style run (see §3)
  RugWatch   — yellow checkbox; when checked, Holders merge RugWatch
               flagged wallets from local DB + cloud URL. When unchecked,
               that section is skipped for this lookup. Choice is remembered
               in this browser.


--------------------------------------------------------------------------------
3. WHAT “QUICK” MEANS
--------------------------------------------------------------------------------

Quick ON (checked)
  - Faster first look.
  - Loads market / Overview-style data.
  - SKIPS slow holder scans, full bundles, and much of the heavy About
    social/news scraping.
  - Use when: you only need price, liquidity, socials, or DexScreener is
    rate-limiting you.

Quick OFF (full Analyze)
  - Full holders, bundles, alerts, and richer About narrative.
  - Can take 30–90+ seconds.
  - Use when: you care about wallet concentration, rugs, or story/news.


--------------------------------------------------------------------------------
4. OVERVIEW TAB
--------------------------------------------------------------------------------

Shows market structure from DexScreener (and Pump.fun when useful):

  Price, market cap / FDV, liquidity, 24h volume, price % changes
  Pair / DEX (e.g. Raydium, Pump.fun, Uniswap)
  Social links from the pair profile (X, Telegram, website)
  Pump.fun status when relevant: still on bonding curve vs graduated

Pump.fun fallback
  - If DexScreener is rate-limited (HTTP 429) and the mint looks like a
    Pump.fun token (often ends in "pump"), the app may still load a basic
    snapshot from Pump.fun’s public coin API.
  - That snapshot is thinner than full DexScreener data.

DexScreener cache
  - Same search/pair is remembered ~2–3 minutes so re-analyzing the same
    mint does not spam DexScreener as hard.


--------------------------------------------------------------------------------
5. HOLDERS TAB & WALLET SCHEME
--------------------------------------------------------------------------------

Purpose
  Show who holds the token (top wallets) and how concentrated supply is.

Display (website)
  - Wallet addresses stay visible and are clickable (open Solscan).
  - Standalone Solscan URL lines are hidden (address is enough).
  - Token holding amounts (balances) are shown in yellow.
  - Supply % uses the priority color bands (see below), including the
    Creator wallet “owns X%” line.
  - Top1 / Top5 / Top10 summary % stay uncolored (summary only).

Solana
  Multi-source fusion when possible:
    Helius / Solana RPC, Rugcheck top holders, Solscan, Birdeye, Pump.fun
  Known LP / AMM / program wallets are labeled and usually treated as
  “not a random human bag” for risk lists.

Other chains (Ethereum, Base, Robinhood, …)
  Market works via DexScreener.
  Full holder fusion is mainly Solana today; you may get an explorer link
  instead of a full holder table.

WALLET PRIORITY SCHEME (used in Alerts, not next to every address)
  Only non-LP wallets with meaningful size are ranked by % of supply:

    Under 2%     → ignored for the “over 2%” list
    2% – 5%      → [low priority]
    5% – under 10% → [medium priority]
    10% – under 15% → [high priority]
    15% or more  → [critical priority]

  In Alerts, those labels appear as a SUBTITLE above groups of wallets
  (e.g. a line “[low priority]” then the wallets under it).

What people often ask
  “Is a 3% wallet bad?”
    → Flagged as low priority for attention; not automatic proof of a rug.
  “Why is the Raydium vault huge?”
    → LP/program wallets are filtered out of the risk list when recognized.
  “Why no holders on Robinhood?”
    → Holder APIs are Solana-focused; use the explorer link for EVM chains.

6. BUNDLES TAB — FULL BUNDLE SCHEME
--------------------------------------------------------------------------------

6.1 What “bundle” means in this app
  A bundle estimate is a HEURISTIC for “how much supply might be controlled by
  coordinated wallets,” not a proof that one person owns those wallets.

  The app looks at top-holder snapshots and risk tags, then estimates a
  total_bundle_pct (% of total supply). That single number drives the
  Alerts “bundle band” messages below.

  This is NOT:
    - A professional sniper-graph product
    - On-chain legal proof of collusion
    - Guaranteed accurate for every launchpad

6.2 Where bundle data comes from (Solana)
  When a full Analyze runs, Bundles may fuse:

    - Helius / RPC top holders
    - Rugcheck (insiders, rugged flag, risk names, networks)
    - Birdeye (optional): bundler / sniper / insider / dev tags, security flags
    - Heuristics on the holder list:
        * Multi-account clusters (one wallet, several large ATAs)
        * Similar-sized non-LP top wallets (look like coordinated buys)
        * Concentration of top wallets excluding known LP / programs
        * Same-slot multi-buy style signals when available (Jito-style / Helius)

  If holders fail, bundles usually cannot run.

  Non-Solana chains (Ethereum, Robinhood, …): market works; full bundle fusion
  is Solana-focused.

6.3 What the Bundles tab shows
  - Summary: estimated total bundle % of supply (when available)
  - Similar-size total %: combined supply of unique wallets that sit in
    similar-size groups (wallets with nearly the same holding size)
  - Signals: e.g. multi-ATA clusters, similar-size groups, Rugcheck insiders,
    Birdeye bundler/sniper tags, rugged flag
  - Lists of suspect / flagged wallets (heuristic)
  - Top10 ex-LP (see §6.3a)
  - Notes that methods are partial

  Display (website) — % color bands (same as Holders priority scheme)
  - Total % bundles, Similar-size total, Suspect total: colored by %
  - Each wallet’s percent holdings in every group is colored the same way:
      multi-account clusters, similar-size group members, insiders, suspects
      (lines look like “wallet  holds X%”)
  - Similar-size group header (right side): “sum X%” = combined holdings of
      all wallets in that group; colored with the same priority bands
  - Group headers (≈ avg % / range) also use the bands
  - Cluster balances ("bal …") shown in yellow
  - Top10 ex-LP stays uncolored (summary only — see §6.3a)

6.3a What “Top10 ex-LP” means
  Top10 ex-LP = combined % of supply held by the top non-LP / non-program
  wallets in the top-holder snapshot (LP vaults and known DEX/program
  accounts are excluded).

  Why it exists
    - Plain Top10 often includes a huge liquidity-pool wallet.
    - That pool % is not the same as “whales who can dump.”
    - Top10 ex-LP answers: how concentrated are the top human-ish wallets
      after removing known LP / program addresses?

  Example
    - Top10 (all accounts): 80% (pool may be most of it)
    - Top10 ex-LP: 35% (top non-LP wallets only)

  On the site
    - Shown on Bundles (and may appear in Logs bundles snapshot).
    - That % is NOT colored with the wallet holding priority bands
      (same idea as Top1 / Top5 / Top10 summary lines).

6.4 Bundle % alert scheme (Alerts tab)
  Alerts use ONE number: estimated bundle % of supply
  (summary.total_bundle_pct; if missing, a fallback from the largest
  similar-size group: count × average %).

  Only ONE bundle-band alert fires — the highest matching tier.

  ┌─────────────────────────┬──────────────────────────────────────────────────┐
  │ Bundle % of supply      │ Alert title / meaning                            │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Under 5%                │ No bundle-% band alert from this check           │
  │                         │ (other alerts can still fire)                    │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ 5% through 20%          │ “Bundle amount low to moderate”                  │
  │ (inclusive band)        │ Severity: medium                                 │
  │                         │ Worth watching; not extreme by itself            │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Above 20% up to 27%     │ “Bundle higher than 20%”                         │
  │ Example: 21%, 26%       │ Severity: high                                   │
  │                         │ High possibility of rug / coordinated dumps      │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Above 27% under 50%     │ “DANGER — bundle 27% or higher”                  │
  │ Example: 28%, 40%       │ Severity: critical                               │
  │                         │ “Most likely rug” style language                 │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ 50% or more             │ “RUG IMMINENT — bundle ≥ 50%”                    │
  │ Example: 50%, 70%       │ Severity: critical                               │
  │                         │ Extremely concentrated coordinated supply        │
  └─────────────────────────┴──────────────────────────────────────────────────┘

  Exact code cutoffs:
    pct < 5.0           → no band alert
    5.0 ≤ pct ≤ 20.0    → low–moderate
    20.0 < pct ≤ 27.0   → higher than 20%
    27.0 < pct < 50.0   → danger 27%+
    pct ≥ 50.0          → rug imminent

6.5 Worked examples
  4% bundle   → no bundle-band alert
  12% bundle  → “low to moderate”
  21% bundle  → “Bundle higher than 20%” (high)
  26% bundle  → still “higher than 20%” (not yet 27% danger)
  30% bundle  → “DANGER — 27% or higher”
  55% bundle  → “RUG IMMINENT — ≥ 50%”

6.6 Bundle vs single-wallet alerts (different systems)
  Bundle %     = estimated coordinated slice of supply (scheme above)
  Wallet %     = each non-LP holder’s own share:

    > 2%  → appears on “wallets over 2%” list with priority band
    > 5%  → also “single holder over 5%” alert
    Priority bands for individual wallets:
      2–5% low · 5–10% medium · 10–15% high · ≥15% critical

  A token can have:
    - Low bundle % but one 12% whale, or
    - High bundle % with many smaller similar wallets

6.7 Extra signals that feed risk (not the same as the % bands)
  - Multi-ATA clusters (one owner, several accounts)
  - Similar-sized top wallets
  - Rugcheck insider_count, rugged=true, risk names (bundle/sniper/insider/…)
  - Birdeye bundler/sniper/insider/dev % tags (if API key set)
  - Mint/freeze authority style security flags (Birdeye)
  These appear as signals/details; the % ladder in §6.4 is what titles the
  main bundle severity alert.

6.8 What people often misunderstand
  “Is 21% a danger alert?”
    → No. 21% is the “higher than 20%” band, not the 27% DANGER title.
  “Under 26% means safe?”
    → No. 21–26% is already high-severity language; under 5% only skips
      the bundle-band alert, not all risk checks.
  “Bundle % is exact on-chain truth?”
    → No. Estimated from snapshots + tags + heuristics.

7. ALERTS TAB (WHAT FIRES AND WHY)
--------------------------------------------------------------------------------

Alerts are “things to look at first.” They combine holders + bundles +
Rugcheck + socials.

7.1 Liquidity unlocked
  - Uses Rugcheck market LP lock stats on pools with meaningful USD size
    (dust pools under ~$5k are ignored).
  - If little LP is locked, the app warns that liquidity may be removable.
  - This is NOT the same as “Pump.fun bonding curve = unlocked” in the
    classic LP sense; curve launches can look “unlocked” in that model.

7.2 Wallets over 2% of supply
  - Full list of non-LP wallets above 2%, grouped by priority (§5).

7.3 Single holder over 5%
  - One non-LP wallet owns more than 5%. Severity higher if ≥ 15%.

7.4 Similar wallets with large combined %
  - Several wallets hold nearly the same size and together are large
    (group of at least 3; combined ≥ 5% or average ≥ 2%).

7.5 Bundle % bands
  - Full scheme in §6 (5–20% / >20–27% / >27% / ≥50%; examples 21% and 26%).

7.6 Socials missing on DexScreener
  - No website / X / Telegram / socials on the pair profile.

7.7 Rug / creator / serial-rug style signals (see §8)

8. RUGCHECK, CREATORS, INSIDERS — HOW FLAGGING WORKS
--------------------------------------------------------------------------------

The app does NOT have a private “serial rugger police database.”
It reads PUBLIC Rugcheck (and optional Birdeye) fields and turns them into
alerts you can read.

8.1 Rugcheck mint flags
  - rugged = true          → strong “this mint was marked rugged” signal
  - risks[] name/description matching words like:
      scam, rug, serial, known scammer, malicious, honeypot,
      creator history, dev sold, bundle, sniper, blacklist, phishing
    → listed as rug-related risks

8.2 Creator / insider style signals
  - Top holders marked insider=true (Rugcheck graph) → listed with % owned
    (often related to creator-linked clusters — still a heuristic).
  - Insider networks (transfer graphs): large networks (e.g. size ≥ 10)
    are called out.
  - known_accounts labels that look scammer-like → flagged if they also
    appear in top holders.

8.3 Birdeye (if BIRDEYE_API_KEY is set)
  - Holder tags: bundler / sniper / insider / dev % of supply
  - Security fields such as mint still enabled, freeze authority, owner
    still holding supply

8.4 What this does NOT mean
  - A flag is not a court verdict.
  - Clean report is not a green light to buy.
  - Missing data is not the same as “safe.”

10. ABOUT TAB — SECTIONS EXPLAINED
--------------------------------------------------------------------------------

About is split into story + community + news + links.

10.1 NARRATIVE (“What this token is about”)
  - Multi-paragraph storyline from official descriptions first, then
    filtered community/web text.
  - Sources can include: CoinGecko, on-chain metadata URI, Pump.fun,
    Birdeye, DexScreener profile, website meta/OG, Jupiter tags,
    Rugcheck meta, etc.
  - Theme labels (politics, AI, animal meme, CTO, …) come from keyword
    matching on name/description/posts — see theme keywords in the app’s
    narrative module if you need the full lists.
  - Confidence: high / medium / low depending on whether real project
    copy was found.
  - If no official description exists, the app refuses to invent a
    backstory.

10.2 Description sources / listing tags / risk notes
  - Short labeled quotes from each string source.
  - Jupiter-style tags when present.
  - Rugcheck risk text (context, not a novel).

10.3 X / COMMUNITY POSTS (social / web score)
  Where: About → “X / COMMUNITY POSTS”

  Fields you see
    Label   bullish | bearish | mixed | neutral | unknown
    Score   about −1.0 (bearish) to +1.0 (bullish)
    Kind    how the score was made (see below)
    Posts analyzed   how many texts were scored
    Sample posts     recent snippets (may include links)

  Kind: text-based (x_text / social_web_text)
    - Reads posts (project X via free mirrors, optional X API, search).
    - Counts positive vs negative keyword hits (lexicon).
    - score = (positive − negative) / (positive + negative)
    - Labels:
        score ≥ +0.25 → bullish
        score ≤ −0.25 → bearish
        else if keywords exist → mixed
        posts but no keywords → neutral

  Kind: market_crowd
    - Used when almost no posts could be fetched.
    - Uses buy vs sell count (weight ~55%) + 24h price change (weight ~45%).
    - Same bullish/bearish cutoffs at ±0.25.
    - This is market flow, NOT reading tweets.

  Positive-leaning sample words (partial list)
    bull, bullish, moon, pump, ath, gem, lfg, buy, long, diamond, hands,
    rally, listing, partnership, organic, gains, …

  Negative-leaning sample words (partial list)
    bear, dump, rug, scam, honeypot, rekt, sell, short, jeet, ngmi,
    fraud, avoid, warning, crash, …

  Important
    - Keyword scoring is crude (slang, sarcasm, and bots break it).
    - Empty posts → score may fall back to market_crowd or unknown.

10.4 PUBLIC NEWS
  Where: About → “PUBLIC NEWS”

  What it is
    - Public headlines and search blurbs that appear to mention this token.
    - Shown with a source tag (e.g. google_news) and a clickable URL when
      available (up to about 12 items).

  Where it comes from
    - Google News RSS (crypto/token-oriented query)
    - DuckDuckGo-style web snippets
    - Other public web hits tagged as news/web in the narrative scrape
    - Must usually mention ticker, name, or contract (filters noise)

  How it relates to the narrative
    - Official descriptions = “what the project claims.”
    - News = “what the public web is saying right now.”
    - News is secondary context, not an official press feed.

  When empty
    - New or obscure tokens often have zero hits.
    - Search APIs can be blocked or empty; empty ≠ “no news on earth.”

  What it is not
    - Not Bloomberg/Reuters paid wires
    - Not verified PR unless it shows up in public search
    - Not a trading signal by itself

10.5 LINKS
  Website, X, Telegram, DexScreener, Solscan / explorers (including
  Robinhood Blockscout when chain is robinhood), etc.

11. RUGGERS (website)
--------------------------------------------------------------------------------

Subtitle: Ruggers

The Ruggers tab sits next to About (before Logs) on the website. It tracks
wallets from your first successful full Analyze of a mint, then lists only
those that later sold out almost completely.

Not financial advice. “Seller” / “swing” labels are heuristics from holder
snapshots, not proof of a rug or intent.

What Ruggers is for
  - Freeze who held what on your FIRST full lookup of a token (from the
    same holder/bundle data that feeds Logs).
  - On later Analyzes of that mint, find wallets that sold ≥99% of that
    first bag — or disappeared from the holder list.
  - Separate similar-size group members from single whales and the creator.
  - Flag buy-backs after a dump as swing (section: Swing traders).
  - Help YOU spot dump wallets and save them to YOUR RugWatch list
    (yellow Upload → cloud, or Export file — see §12).

How tracking starts (first lookup)
  After a successful full Analyze (Quick OFF) with holders:

  1. Top non-LP holders from the Holders snapshot are stored as a baseline.
  2. Similar-size wallets from Bundles (similar-size groups) are included.
  3. Creator wallet (when known) is included.
  4. Each wallet’s first supply % (and balance when known) is frozen.
  5. That freeze is what later “sold % of holdings” is measured against.

  Quick mode does not start or update Ruggers (no full holder scan).

Does the number of lookups before they sell matter?
  NO. Lookup count does not matter.

  - Sale is always measured against the FIRST full lookup bag only.
  - It does not matter if the dump shows up on the 2nd, 3rd, 10th, or
    50th full Analyze of the same mint.
  - Creator, similar wallets, and single wallets from that first freeze
    are all treated the same way: whenever a later full Analyze first
    shows they sold ≥99% (or dropped off the list), they count as sellers.
  - Intermediate lookups do not reset or weaken tracking. You do not need
    to “catch” the sell on a particular lookup number.

  Example
    Lookup 1: wallet holds 5% (frozen as first bag)
    Lookups 2–9: still holds ~5% → not listed as seller
    Lookup 10: holds 0.02% or gone from the list → SELLER
      (same rule as if they had sold on lookup 2)

What “sold ≥99%” means
  A wallet is listed only if it dumped at least 99% of its FIRST-lookup bag:

    remaining_first_bag ≤ 1% of first bag
    OR the wallet is no longer on the holder list (treated as sold 100%)

  Examples
    First hold 5.00% of supply → still listed with 0.04% → seller (~99.2% sold)
    First hold 5.00% → still listed with 0.10% → NOT listed (only 98% sold)
    First hold 5.00% → gone from top holders → seller (not listed anymore)
    First hold 5.00% → sold out, then later holds 1% again → swing
    First hold 5.00% → still holding until 10th lookup, then sold ≥99%
      → still a seller (lookup number does not matter)

  Wallets that never sold ≥99% are ignored (not shown).

Sections on the Ruggers tab
  1. Creator (sold ≥99%)
       Creator only, and only if they sold ≥99% of first bag or left the list
       on any later full Analyze (2nd, 10th, … — count does not matter).
  2. Similar wallets (sellers)
       Wallets that were in a similar-size group on first lookup and later
       sold ≥99% / dropped off the list (any later lookup). Listed as SELLER.
  3. Single wallets (sellers)
       Individual top holders (not in a similar group) that sold ≥99% or
       are no longer listed on any later full Analyze. Listed as SELLER.
  4. Swing traders
       Wallets that previously met the seller rule, then bought back on a
       later lookup. Labeled swing (not seller).

Labels
  seller  — sold ≥99% of first bag or dropped off the holder list
  swing   — was a seller, then reappeared with holdings (buy-back)

Buttons
  Refresh     — redraw the panel from browser storage
  Clear mint  — delete tracking for the current / selected mint only
  Clear all   — wipe all Ruggers tracking on this browser

Per-section actions (Creator / Similar / Single only — not Swing)
  Export (gray)  — download JSON or one-address-per-line file for manual
                   import in RugWatch “Upload manual wallets”
  Upload (yellow)— send that section’s wallets to YOUR RugWatch website:
                   1) import into RugWatch local DB
                   2) Push cloud → GitHub (cloud wallet count increases)
                   Requires RugWatch running (default http://127.0.0.1:8790)
                   with GITHUB_TOKEN set. Configure rugwatchUrl in config.js.

Where it lives
  Website only: browser localStorage key adtc_ruggers_track
  (separate from Logs). Not uploaded to the server. Not shared across
  browsers / devices. Clearing site data resets it.

  Multiple mints: use the dropdown on the Ruggers tab to switch.

Typical workflow
  1. Full Analyze a new mint (Quick off) → baseline freezes (may show
     empty seller sections — nothing has dumped yet).
  2. Later, Analyze the same mint again (2nd, 3rd, 10th… — any later full
     run; how many times you looked first does not change the rule).
  3. Open Ruggers → see who sold ≥99% of their first bag, by section.
  4. If a dumper buys back, they move to Swing traders.
  5. Click yellow Upload on Creator / Similar / Single (with RugWatch
     website running) so those wallets enter your local + cloud list —
     or Export and import in RugWatch manually (see §12).

What Ruggers is not
  - Not a live mempool dump alert
  - Not proof wallets are the same person or a “rug team”
  - Not RugWatch itself (RugWatch is the separate wallet DB / website)
  - Not updated by Quick-only searches
  - Not a substitute for Solscan trade history (top-holder snapshots only)
  - Yellow Upload needs YOUR RugWatch instance; it does not write to a
    shared public police DB for everyone


--------------------------------------------------------------------------------
12. SAVING SELLER WALLETS TO RUGWATCH (LOCAL + CLOUD)
--------------------------------------------------------------------------------

Subtitle: RugWatch (local DB + GitHub cloud)

Read this even if you only use the website.

Two different things (do not mix them up)
  Ruggers (this site’s tab)
    - Temporary dump spotting for mints you analyze in THIS browser.
    - Stored only in browser localStorage (not on the ATC host by default).
    - Does NOT by itself host a global serial-rugger DB for all users.

  RugWatch (separate app / website)
    - YOUR serial-rugger / flagged-wallet warehouse.
    - Local: SQLite data/rugwatch.db (+ auto-shards rugwatch_002.db… at
      100k wallets per file).
    - Cloud: GitHub files data/wallets_cloud.json (+ _002…) and
      data/wallets_index.json after Push cloud.
    - ATC reads local and/or cloud when flagging Holders (see §12b).

Ways to grow YOUR RugWatch list from Ruggers
  A) Yellow Upload (recommended when RugWatch website is running)
     Ruggers → Creator / Similar / Single → Upload
     → RugWatch imports wallets → Push cloud
     → blue cloud pill / GitHub index total increases
  B) Export → file → RugWatch “Upload manual wallets” → Push cloud
  C) Copy addresses by hand → RugWatch Add wallet → Push cloud

What the ATC host (e.g. Render) will NOT do for everyone
  - Will NOT store one shared public “serial rugger police” DB for all users
  - Will NOT replace your private RugWatch cloud/repo
  - Will NOT sync your browser Ruggers list to other users or devices

Why this matters
  - Ruggers “seller” is a heuristic (≥99% of first bag / left list) — not a
    court verdict. YOU decide what enters long-term RugWatch.
  - After wallets are in cloud (and ATC has RUGWATCH_WALLETS_URL), later
    Analyzes can flag them on Holders (with RugWatch checkbox ON).

Summary
  Ruggers  = temporary browser tool to spot dumps on a mint.
  RugWatch = YOUR flagged-wallet DB (local + optional GitHub cloud).
  Yellow Upload / Export / Add wallet = how YOUR list grows.
  ATC flags = read-only merge of local + cloud (see next section).


--------------------------------------------------------------------------------
12b. WHERE ATC GETS FLAGGED WALLETS (LOCAL, CLOUD, OR BOTH)
--------------------------------------------------------------------------------

Subtitle: Flagged wallets sources

When Analyze runs with the RugWatch checkbox ON, Holders can show
“FLAGGED WALLETS (RugWatch)” from:

  1) Local  — all RugWatch SQLite shards found (rugwatch.db, _002, …)
  2) Cloud  — JSON from env RUGWATCH_WALLETS_URL
              Prefer the multi-shard index:
              https://raw.githubusercontent.com/YOU/RugWatch/main/data/wallets_index.json
              (falls back to single wallets_cloud.json)

Each match is tagged:
  [local]  — only in local DB
  [cloud]  — only in the cloud list
  [both]   — in local and cloud

On Render (live ATC)
  Usually cloud only (no RugWatch folder on the server).
  Set RUGWATCH_WALLETS_URL to the index (or cloud file) and redeploy.

On your PC (local ATC + RugWatch)
  Can be local + cloud at once if both exist.

Checkbox OFF
  Skip RugWatch merge entirely for that Analyze (no flags from those lists).

13. LOGS
--------------------------------------------------------------------------------

Subtitle: Logs

The Logs tab is a private log of tokens you have already searched.
It sits next to Overview, Holders, Bundles, Alerts, Maps, About, and Ruggers
on the desktop app and the website. Use it to look back at recent analyzes
without typing every mint again.

  Desktop: saved in history_log.json next to the app.
  Website: saved in this browser only (localStorage).

What Logs is for
  - See a list of your last token searches in one place.
  - Review market, holder, bundle, and alert snapshots from those runs.
  - Download the list to a file for your own notes.

When a search is saved
  - A row is added only after a successful full Analyze finishes
    (status shows OK / full data loaded).
  - Failed searches (not found, errors) are not saved.
  - Quick or partial loads wait until the full run completes before
    they appear in Logs.
  - Searching the same token again adds another row (each run is kept
    as its own entry).

What each entry shows
  A snapshot from the moment you searched — not live prices:

    · When you searched (UTC)
    · Symbol, name, mint address, chain
    · DEX / pool when known
    · Price, market cap, liquidity, 24h volume and change
    · Holder concentration and Top1 / Top5 / Top10 %
    · Bundle risk and share %
    · How many top-priority alerts fired
    · Pump.fun status when known
    · Pair link when known

  Plus frozen text “screenshots” saved at lookup time:

    · HOLDERS SNAPSHOT — the Holders report as it looked when you
      Analyzed (top wallets, %, creator, flags when available).
      Long reports may be truncated so 20 entries still fit in storage.
    · BUNDLES SNAPSHOT — the Bundles report as it looked at that same
      moment (totals, risk, suspect signals when available). Also may
      be truncated for space.

  These freeze at search time. They do not update later. To get a fresh
  holders/bundles list, run Analyze again and open Holders / Bundles.

How many entries
  - Logs keeps up to 20 searches.
  - Newest is at the top.
  - When you go past 20, the oldest entry is removed automatically.

Where it lives
  Desktop
  - Saved as history_log.json next to the app.
  Website
  - Saved in your browser only (not on the server, not shared with others).
  - Clearing site data / another browser / private mode starts empty.
  - Different from the small search-bar dropdown (desktop only).

Buttons
  Refresh
    Reloads the Logs list from storage (desktop: history_log.json on
    disk; website: this browser’s saved data). Use if the tab looks
    stale after Analyze, or after you reopened the app. Does not
    re-scan tokens and does not call APIs.

  Clear
    After you confirm, deletes ALL Logs entries for this app/browser.
    Cannot be undone. Desktop empties history_log.json; website clears
    localStorage. Does not affect Overview/Holders for the token you
    currently have open.

  Download
    Saves the current Logs list to a file you choose.
    · .txt — readable text (same layout as the tab, including
      holders/bundles snapshots)
    · .json — full structured data for backup or tools
    Does not delete Logs from the app. Does not re-fetch market data.

How to use it
  1. Analyze a token and wait until the run finishes successfully.
  2. Open the Logs tab — that token should be first on the list, with
     holders and bundles snapshots under the summary.
  3. Keep analyzing other tokens (up to 20 kept).
  4. Use Download if you want a copy of the log.
  5. Use Clear only if you want an empty Logs list.
  6. Use Refresh if the list did not update after Analyze.

What Logs is not
  - Not a live price feed or live holders feed.
  - Not a camera image — “snapshot” means a frozen text copy of the
    Holders / Bundles report at Analyze time.
  - Not a full export of every report field beyond what was stored
    (use Export JSON on desktop for the full current report).
  - Not a safety rating or “safe list.” Past results are for your
    reference only.

15. COMMON QUESTIONS
--------------------------------------------------------------------------------

Q: Can visitors add their own Helius keys?
A: No. Keys stay on the server. Gear = site passcode only (if enabled).

Q: Why DexScreener 429 / bad response?
A: Public rate limits. Wait, use Quick, use full mint, enable site passcode
   so fewer people spam Analyze. Cache reduces repeat hits.

Q: Do view counts reset daily?
A: Total views do not reset daily. “Unique today” does. Free hosting may
   reset totals after redeploy.

Q: Why is LP unlocked on a Pump.fun coin?
A: LP-lock check is classic pool lock %, not a full bonding-curve model.
   On desktop Alerts, that LP-unlock warning is skipped for Pump.fun /
   PumpSwap pools (and classic “…pump” mints) because those venues do
   not map cleanly to locked LP pools.

Q: Does a clean Alerts tab mean safe?
A: No. Absence of flags ≠ safety.

Q: Where did my Logs go after reinstall or moving the app?
A: Desktop Logs are saved in history_log.json next to the app. Website
   Logs live in the browser only. If you move the app folder or clear
   site data, the list may be gone. Use Download to keep a backup.

Q: Is there a serial-rugger database? Who creates it?
A: Yes — RugWatch on your desktop is that database (local SQLite, wallets
   you flag). The website Ruggers tab does not create or update it. Copy
   seller addresses from Ruggers and add them in RugWatch yourself (§12).
   The live website host does not keep a shared serial-rugger DB for all
   users.

Q: If a wallet sells on my 10th lookup, do they still count as a seller?
A: Yes. Lookup count does not matter. Sales are always measured against
   the first full Analyze bag (see §11).
