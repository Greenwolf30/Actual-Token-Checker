--------------------------------------------------------------------------------
1. WHAT THIS APP IS
--------------------------------------------------------------------------------

A research tool for crypto tokens (especially Solana, plus other chains like
Ethereum, Base, and Robinhood Chain). You paste a contract or ticker; the app
pulls public market and risk data and shows it in tabs:

  Overview · Holders · Bundles · Alerts · Maps · About · Ruggers · Logs

It does NOT trade for you, custody funds, or guarantee a token is safe.

What this guide covers vs RugWatch
  THIS DOC (Token Checker / ATC)
    - Analyze UI, tabs, Quick mode, Ruggers lanes, Logs, Alerts, Maps, About
    - How ATC *reads* Flagged (RugWatch) and shows hold %
    - How Ruggers Upload/Export *sends* wallets into YOUR RugWatch
  RUGWATCH DOC (separate app — RugCheck Documentation.md)
    - Building the wallet list, risk_score 0–100, Monitor, Push/Pull cloud
    - Scan mint, Add/Upload wallets, local DB shards, capacity
  Do not paste full RugWatch manuals into ATC docs (and vice versa).

2. HOW TO USE THE SITE
--------------------------------------------------------------------------------

  1. Open the site (wait ~1 minute if Free hosting is waking up).
  2. Paste a mint address, 0x address, or name/ticker.
  3. Optional: pick a chain (auto / solana / robinhood / ethereum / …).
  4. Optional: check Quick (faster, less detail — see §3).
  5. Optional: check RugWatch (yellow) — include flagged wallets on Analyze
     (default ON). Uncheck to ignore local + cloud RugWatch lists for this run.
  6. Click Analyze.
  7. Read the tabs. Click blue links to open explorers / X / news.


Navigation bar (compact)
  [logo]  [RugWatch]  |  Docs | GitHub | Stats API | Badge

  Logo stays far left.
  RugWatch (yellow letters) sits immediately to the right of the logo and
    opens the RugWatch website (default http://127.0.0.1:8790/ — set
    rugwatchUrl in web/config.js when you have a public URL).
  Docs → /docs.html on this website (loads web/documentation.txt)
  Also: /documentation.txt (raw text) · /docs (same as docs.html)
  GitHub → https://github.com/Greenwolf30/Actual-Token-Checker
  Footer “Documentation” link opens the on-site guide (not only GitHub).
  Analyze is the search button under the title.

Analyze options (next to the search form)
  quick      — faster market-only style run (see §3)
  RugWatch   — yellow checkbox; when checked, Holders merge RugWatch
               flagged wallets from local DB + cloud URL. When unchecked,
               that section is skipped for this lookup. Choice is remembered
               in this browser.

3. WHAT “QUICK” MEANS
--------------------------------------------------------------------------------

Quick ON (checked)
  - Faster first look.
  - Loads market / Overview-style data.
  - SKIPS slow holder scans, full bundles, and much of the heavy About
    social/news scraping.

Quick OFF (full Analyze)
  - Full holders, bundles, alerts, and richer About narrative.
  - Can take 30–90+ seconds.
  - Use when: you care about wallet concentration, rugs, or story/news.

4. OVERVIEW TAB
--------------------------------------------------------------------------------

Section titles (website): ── MARKET ── · ── PUMP.FUN ── · ── INITIAL MARKET CAP ──
  · ── ALL-TIME HIGH ── · ── SOCIALS ──

Shows market structure from DexScreener (and Pump.fun when useful):

  Price, market cap / FDV, liquidity, 24h volume, price % changes
  Pair / DEX (e.g. Raydium, Pump.fun, Uniswap)
  Social links from the pair profile (X, Telegram, website)
  Pump.fun status when relevant: still on bonding curve vs graduated
  Initial MC estimate and ATH MC estimate when history is available

Pump.fun fallback
  - If DexScreener is rate-limited (HTTP 429) and the mint looks like a
    Pump.fun token (often ends in "pump"), the app may still load a basic
    snapshot from Pump.fun’s public coin API.
  - That snapshot is thinner than full DexScreener data.

DexScreener / Analyze cache (server, in-memory)
  - Same DexScreener search/pair is remembered ~2–3 minutes.
  - Full Analyze result for the same mint is cached ~90s (Quick ~45s).
  - Concurrent Analyzes of the same mint share one live fetch (single-flight).
  - Per-IP: max ~6 Analyzes / 60s and one Analyze at a time (protects shared
    host IP from rate limits). Env: ANALYZE_CACHE_TTL_FULL, ANALYZE_RATE_MAX.
  - Mint addresses use targeted token-pairs / Pump before broad DexScreener
    search to cut free-API pings.
  - DexScreener soft budget (~400 calls/hour) + ~90s cooldown after HTTP 429;
    Solana mints then fall back to Raydium API v3 (pools/price). Env:
    DEXSCREENER_HOUR_BUDGET, DEXSCREENER_COOLDOWN_SEC.
  - Pre-bond *pump tokens: native Pump.fun API is primary (price/mcap/curve);
    DexScreener/Raydium are not required while still on the bonding curve.
  - Multi-source stack (use what responds; no single point of failure):
      Market: Pump (prebond) → DexScreener → Raydium → Birdeye → Rugcheck → Pump
      Holders: Helius/RPC + Rugcheck + Solscan + Birdeye (fused in parallel)
      LP labels: known programs + Pump curve/pool PDAs

5. HOLDERS TAB & WALLET SCHEME
--------------------------------------------------------------------------------

Website section markers (dim green):
  ── HOLDERS / WALLETS ── · ── TOTAL WALLETS ── · ── CONCENTRATION ──
  · ── FLAGS ── · ── TOP HOLDERS ── · ── MULTI-ACCOUNT CLUSTERS ──
  · ── FLAGGED WALLETS (RugWatch) ──

  Total wallets by source when available (Pump.fun / Birdeye / DexScreener /
  Solscan). Top holders with supply % priority colors. Creator, mint/freeze
  authority when known. RugWatch flagged still-holding / previously-holding
  when checkbox ON (see §12b).

Purpose
  Show who holds the token (top wallets) and how concentrated supply is.

Display (website)
  - Wallet addresses stay visible and are clickable (open Solscan).
  - Standalone Solscan URL lines are hidden (address is enough).
  - Token holding amounts (balances) are shown in yellow.
  - Supply % uses the priority color bands (see below), including the
    Creator wallet “owns X%” line.
  - Top1 / Top5 / Top10 summary % stay uncolored (summary only).

Solana
  Multi-source fusion when possible:
    Helius / Solana RPC, Rugcheck top holders, Solscan, Birdeye, Pump.fun
  Known LP / AMM / program wallets are labeled and usually treated as
  “not a random human bag” for risk lists.

Other chains (Ethereum, Base, Robinhood, …)
  Market works via DexScreener.
  Full holder fusion is mainly Solana today; you may get an explorer link
  instead of a full holder table.

WALLET PRIORITY SCHEME (used in Alerts, not next to every address)
  Only non-LP wallets with meaningful size are ranked by % of supply:

    Under 2%     → ignored for the “over 2%” list
    2% – 5%      → [low priority]
    5% – under 10% → [medium priority]
    10% – under 15% → [high priority]
    15% or more  → [critical priority]

  In Alerts, those labels appear as a SUBTITLE above groups of wallets
  (e.g. a line “[low priority]” then the wallets under it).

What people often ask
  “Is a 3% wallet bad?”
    → Flagged as low priority for attention; not automatic proof of a rug.
  “Why is the Raydium vault huge?”
    → LP/program wallets are filtered out of the risk list when recognized.
  “Why no holders on Robinhood?”
    → Holder APIs are Solana-focused; use the explorer link for EVM chains.

6. BUNDLES TAB — FULL BUNDLE SCHEME
--------------------------------------------------------------------------------

6.1 What “bundle” means in this app
  A bundle estimate is a HEURISTIC for “how much supply might be controlled by
  coordinated wallets,” not a proof that one person owns those wallets.

  The app looks at top-holder snapshots and risk tags, then estimates a
  total_bundle_pct (% of total supply). That single number drives the
  Alerts “bundle band” messages below.

  This is NOT:
    - A professional sniper-graph product
    - On-chain legal proof of collusion
    - Guaranteed accurate for every launchpad

6.2 Where bundle data comes from (Solana)
  When a full Analyze runs, Bundles may fuse:

    - Helius / RPC top holders
    - Rugcheck (insiders, rugged flag, risk names, networks)
    - Birdeye (optional): bundler / sniper / insider / dev tags, security flags
    - Heuristics on the holder list:
        * Multi-account clusters (one wallet, several large ATAs)
        * Similar-sized non-LP top wallets (look like coordinated buys)
        * Concentration of top wallets excluding known LP / programs
        * Same-slot multi-buy style signals when available (Jito-style / Helius)

  If holders fail, bundles usually cannot run.

  Non-Solana chains (Ethereum, Robinhood, …): market works; full bundle fusion
  is Solana-focused.

6.3 What the Bundles tab shows
  Section titles use dim-green ── TITLE ── markers (website).

  - Summary: estimated total bundle % of supply (when available)
  - Similar-size total %: combined supply of unique wallets in similar-size groups
  - ── MULTI-ACCOUNT CLUSTERS ──  one owner, several large ATAs
  - ── SIMILAR-SIZE GROUPS ──     nearly same bag size among top wallets
  - ── INSIDER-FLAGGED ──         Rugcheck insider tags (not RugWatch)
  - ── SHARED SOL FUNDER ──       1-hop common SOL funder (Helius when available)
  - ── LAUNCH-WINDOW ──           same-slot multi-buys at launch (time window)
  - ── SUSPECT WALLETS ──         union of signals (heuristic)
  - Signals: multi-ATA, similar-size, insiders, funder, launch, Birdeye tags, rugged
  - Top10 ex-LP (see §6.3a)
  - Notes that methods are partial

  Display (website) — % color bands (same as Holders priority scheme)
  - Total % bundles, Similar-size total, Suspect total: colored by %
  - Each wallet’s percent holdings in every group is colored the same way
      (lines look like “wallet  holds X%”)
  - Similar-size group header (right side): “sum X%” = combined group holdings
  - Group headers (≈ avg % / range) also use the bands
  - Cluster balances ("bal …") shown in yellow
  - Top10 ex-LP stays uncolored (summary only — see §6.3a)

  Launch-window vs other signals
    | Section              | What it tracks                              |
    |----------------------|---------------------------------------------|
    | Launch-window        | Many buys in the same slot / short launch   |
    |                      | time window (not “same bag size”)           |
    | Similar-size         | Same holding size, not necessarily same time|
    | Multi-account        | Same owner, multiple large token accounts   |
    | Shared SOL funder    | Same funding parent (1-hop SOL), not slot   |
    | Insider (Rugcheck)   | Rugcheck insider flag on top holders        |

6.3a What “Top10 ex-LP” means
  Top10 ex-LP = combined % of supply held by the top non-LP / non-program
  wallets in the top-holder snapshot (LP vaults and known DEX/program
  accounts are excluded).

  Why it exists
    - Plain Top10 often includes a huge liquidity-pool wallet.
    - That pool % is not the same as “whales who can dump.”
    - Top10 ex-LP answers: how concentrated are the top human-ish wallets
      after removing known LP / program addresses?

  Example
    - Top10 (all accounts): 80% (pool may be most of it)
    - Top10 ex-LP: 35% (top non-LP wallets only)

  On the site
    - Shown on Bundles (and may appear in Logs bundles snapshot).
    - That % is NOT colored with the wallet holding priority bands
      (same idea as Top1 / Top5 / Top10 summary lines).

6.4 Bundle % alert scheme (Alerts tab)
  Alerts use ONE number: estimated bundle % of supply
  (summary.total_bundle_pct; if missing, a fallback from the largest
  similar-size group: count × average %).

  Only ONE bundle-band alert fires — the highest matching tier.

  ┌─────────────────────────┬──────────────────────────────────────────────────┐
  │ Bundle % of supply      │ Alert title / meaning                            │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Under 5%                │ No bundle-% band alert from this check           │
  │                         │ (other alerts can still fire)                    │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ 5% through 20%          │ “Bundle amount low to moderate”                  │
  │ (inclusive band)        │ Severity: medium                                 │
  │                         │ Worth watching; not extreme by itself            │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Above 20% up to 27%     │ “Bundle higher than 20%”                         │
  │ Example: 21%, 26%       │ Severity: high                                   │
  │                         │ High possibility of rug / coordinated dumps      │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ Above 27% under 50%     │ “DANGER — bundle 27% or higher”                  │
  │ Example: 28%, 40%       │ Severity: critical                               │
  │                         │ “Most likely rug” style language                 │
  ├─────────────────────────┼──────────────────────────────────────────────────┤
  │ 50% or more             │ “RUG IMMINENT — bundle ≥ 50%”                    │
  │ Example: 50%, 70%       │ Severity: critical                               │
  │                         │ Extremely concentrated coordinated supply        │
  └─────────────────────────┴──────────────────────────────────────────────────┘

  Exact code cutoffs:
    pct < 5.0           → no band alert
    5.0 ≤ pct ≤ 20.0    → low–moderate
    20.0 < pct ≤ 27.0   → higher than 20%
    27.0 < pct < 50.0   → danger 27%+
    pct ≥ 50.0          → rug imminent

6.5 Worked examples
  4% bundle   → no bundle-band alert
  12% bundle  → “low to moderate”
  21% bundle  → “Bundle higher than 20%” (high)
  26% bundle  → still “higher than 20%” (not yet 27% danger)
  30% bundle  → “DANGER — 27% or higher”
  55% bundle  → “RUG IMMINENT — ≥ 50%”

6.6 Bundle vs single-wallet alerts (different systems)
  Bundle %     = estimated coordinated slice of supply (scheme above)
  Wallet %     = each non-LP holder’s own share:

    > 2%  → appears on “wallets over 2%” list with priority band
    > 5%  → also “single holder over 5%” alert
    Priority bands for individual wallets:
      2–5% low · 5–10% medium · 10–15% high · ≥15% critical

  A token can have:
    - Low bundle % but one 12% whale, or
    - High bundle % with many smaller similar wallets

6.7 Extra signals that feed risk (not the same as the % bands)
  - Multi-ATA clusters (one owner, several accounts)
  - Similar-sized top wallets
  - Rugcheck insider_count, rugged=true, risk names (bundle/sniper/insider/…)
  - Birdeye bundler/sniper/insider/dev % tags (if API key set)
  - Mint/freeze authority style security flags (Birdeye)
  These appear as signals/details; the % ladder in §6.4 is what titles the
  main bundle severity alert.

6.8 What people often misunderstand
  “Is 21% a danger alert?”
    → No. 21% is the “higher than 20%” band, not the 27% DANGER title.
  “Under 26% means safe?”
    → No. 21–26% is already high-severity language; under 5% only skips
      the bundle-band alert, not all risk checks.
  “Bundle % is exact on-chain truth?”
    → No. Estimated from snapshots + tags + heuristics.


--------------------------------------------------------------------------------
7. ALERTS TAB (WHAT FIRES AND WHY)
--------------------------------------------------------------------------------

Alerts are “things to look at first.” They combine holders + bundles +
Rugcheck + socials.

7.1 Liquidity unlocked
  - Uses Rugcheck market LP lock stats on pools with meaningful USD size
    (dust pools under ~$5k are ignored).
  - If little LP is locked, the app warns that liquidity may be removable.
  - This is NOT the same as “Pump.fun bonding curve = unlocked” in the
    classic LP sense; curve launches can look “unlocked” in that model.

7.2 Wallets over 2% of supply
  - Full list of non-LP wallets above 2%, grouped by priority (§5).

7.3 Single holder over 5%
  - One non-LP wallet owns more than 5%. Severity higher if ≥ 15%.

7.4 Similar wallets with large combined %
  - Several wallets hold nearly the same size and together are large
    (group of at least 3; combined ≥ 5% or average ≥ 2%).

7.5 Bundle % bands
  - Full scheme in §6 (5–20% / >20–27% / >27% / ≥50%; examples 21% and 26%).

7.6 Socials missing on DexScreener
  - No website / X / Telegram / socials on the pair profile.

7.7 Rug / creator / serial-rug style signals (see §8)


--------------------------------------------------------------------------------
8. RUGCHECK, CREATORS, INSIDERS — HOW FLAGGING WORKS
--------------------------------------------------------------------------------

The app does NOT have a private “serial rugger police database.”
It reads PUBLIC Rugcheck (and optional Birdeye) fields and turns them into
alerts you can read.

8.1 Rugcheck mint flags
  - rugged = true          → strong “this mint was marked rugged” signal
  - risks[] name/description matching words like:
      scam, rug, serial, known scammer, malicious, honeypot,
      creator history, dev sold, bundle, sniper, blacklist, phishing
    → listed as rug-related risks

8.2 Creator / insider style signals
  - Top holders marked insider=true (Rugcheck graph) → listed with % owned
    (often related to creator-linked clusters — still a heuristic).
  - Insider networks (transfer graphs): large networks (e.g. size ≥ 10)
    are called out.
  - known_accounts labels that look scammer-like → flagged if they also
    appear in top holders.

8.3 Birdeye (if BIRDEYE_API_KEY is set)
  - Holder tags: bundler / sniper / insider / dev % of supply
  - Security fields such as mint still enabled, freeze authority, owner
    still holding supply

8.4 What this does NOT mean
  - A flag is not a court verdict.
  - Clean report is not a green light to buy.
  - Missing data is not the same as “safe.”

--------------------------------------------------------------------------------
10.0 WHERE THE DATA COMES FROM (SOURCE MAP)
--------------------------------------------------------------------------------

A) Official / listing copy (“what the project claims”) — coin_facts
  Used first for the storyline and “official description.”
  Best-effort, short timeouts; failures are soft (no crash).

  | Source                         | What we take                          |
  |--------------------------------|---------------------------------------|
  | CoinGecko contract endpoint    | Description, categories, links        |
  | On-chain metadata URI (JSON)   | Metaplex-style description / links    |
  | Pump.fun coin JSON API         | Description, name, twitter, telegram, |
  |   (not HTML scrape of pump.fun)| website, metadata URI when present    |
  | Birdeye (when available)       | Description / extensions              |
  | DexScreener pair / profile     | Description, socials, websites        |
  | CoinMarketCap (when found)     | Description / links                   |
  | Jupiter / token lists          | Tags (meme, AI, …)                    |
  | Solscan (when used)            | Metadata / links                      |
  | Rugcheck                       | Meta description, risk text notes     |
  | Project website OG / meta      | Page description when a site is linked|

  Pump.fun detail
  - About does NOT scrape the HTML page https://pump.fun/<mint>.
  - It calls the same style of JSON the site uses, e.g.:
      frontend-api.pump.fun/coins/{mint}
      frontend-api-v3.pump.fun/coins/{mint}
  - That JSON supplies description + social URLs when the API responds.
  - Cloudflare / API downtime → no Pump data for that run (normal).

  Raydium
  - Used for MARKET pairs/price fallback when DexScreener is weak — NOT for
    About narrative copy (no description/social feed from Raydium).

B) Community / web chatter — social_sources (secondary only)
  | Source                    | Role                                      |
  |---------------------------|-------------------------------------------|
  | X / Twitter               | Project handle posts + $ticker search     |
  | Reddit public search JSON | Recent posts mentioning ticker/name       |
  | Google News RSS           | Headlines                                 |
  | LinkedIn                  | Profile links + site:linkedin.com search |
  | DuckDuckGo HTML           | Web blurbs (may tag TikTok/IG/LinkedIn)   |
  | Linked social URLs        | IG/TikTok/X/Telegram/LinkedIn on profile  |

  X / Twitter detail
  - Primary handle usually from DexScreener socials (and extra X links).
  - Optional X API (if X_BEARER_TOKEN set) + free Nitter-style mirrors.
  - Pump.fun twitter field: stored as a linked social + coin links.
  - Full X timeline/bio scrape uses the project handle — mainly when Dex
    (or merged socials) provide it. Pump-only X is always noted as a link;
    posts/bio may not run if that handle is never set as twitter_handle.
  - Does NOT resolve “creator wallet → personal X account” by itself.

  Reddit / LinkedIn / TikTok / Instagram
  - No free full-feed APIs for IG/TikTok/LinkedIn.
  - Reddit: public search.json (best-effort).
  - LinkedIn: profile links on the token + DuckDuckGo site:linkedin.com.
  - IG/TikTok: only public links and search mentions, not full scrapes.

C) What About does NOT use
  - RugWatch local/cloud wallet DB (that is Holders / Alerts / Ruggers)
  - Full holder scans
  - Paid news terminals (Bloomberg, etc.)
  - Invented backstory when no official copy exists

Priority rule
  Official/listing descriptions rank above community/web chatter.
  Community snippets are filtered to mentions of this ticker, name, or
  contract when possible.

--------------------------------------------------------------------------------
10.1 NARRATIVE (“What this token is about”)
--------------------------------------------------------------------------------

  - Multi-paragraph storyline from official descriptions first, then
    filtered community/web text.
  - Theme labels (politics, AI, animal meme, CTO, …) come from keyword
    matching on name/description/posts.
  - Confidence: high / medium / low depending on whether real project
    copy was found (and which source won).
  - “Why people seem interested” / Key hooks may include:
      · official description present (stated purpose/story: <text>)
      · listing categories / tags
      · Reddit / LinkedIn / news / Pump.fun metadata when detected
  - If no official description exists, the app refuses to invent a
    backstory (market tags + filtered chatter only).

--------------------------------------------------------------------------------
10.2 Description sources / listing tags / risk notes
--------------------------------------------------------------------------------

  - Short labeled quotes from each string source (e.g. [pumpfun], [coingecko]).
  - Jupiter-style tags when present.
  - Rugcheck risk text (context, not a novel).

--------------------------------------------------------------------------------
10.3 X / COMMUNITY POSTS (social / web score)
--------------------------------------------------------------------------------

  Where: About → “X / COMMUNITY POSTS”

  Compact meta (always with values or n/a)
    Tone: <label> · score <n|n/a> · kind <kind> · posts analyzed <n>
    Sources: <list or n/a>
    Handle / Profile when known
    Summary / Note when present
    Then: Recent X posts (or empty-message)

  Fields
    Label   bullish | bearish | mixed | neutral | unknown
    Score   about −1.0 (bearish) to +1.0 (bullish)
    Kind    how the score was made (see below)
    Posts analyzed   how many texts were scored
    Sample posts     recent snippets (may include links)

  Kind: text-based (x_text / social_web_text)
    - Reads posts (project X via free mirrors, optional X API, $ticker search).
    - Counts positive vs negative keyword hits (lexicon).
    - score = (positive − negative) / (positive + negative)
    - Labels:
        score ≥ +0.25 → bullish
        score ≤ −0.25 → bearish
        else if keywords exist → mixed
        posts but no keywords → neutral

  Kind: market_crowd
    - Used when almost no posts could be fetched.
    - Uses buy vs sell count (weight ~55%) + 24h price change (weight ~45%).
    - Same bullish/bearish cutoffs at ±0.25.
    - This is market flow, NOT reading tweets.

  Positive-leaning sample words (partial list)
    bull, bullish, moon, pump, ath, gem, lfg, buy, long, diamond, hands,
    rally, listing, partnership, organic, gains, …

  Negative-leaning sample words (partial list)
    bear, dump, rug, scam, honeypot, rekt, sell, short, jeet, ngmi,
    fraud, avoid, warning, crash, …

  Important
    - Keyword scoring is crude (slang, sarcasm, and bots break it).
    - Empty posts → score may fall back to market_crowd or unknown.
    - Pump.fun-linked X is a social link; timeline scrape needs a project
      handle (usually DexScreener) to populate sample posts.

11. RUGGERS (website)
--------------------------------------------------------------------------------

Subtitle: Ruggers

The Ruggers tab sits next to About (before Logs) on the website. It freezes
a holder/bundle baseline on your FIRST full Analyze of a mint, then on later
Analyzes lists only wallets that sold ≥99% of that first bag (or left the
holder list). Buy-backs go to Swing with the same origin label.

Not financial advice. “Seller” / “swing” labels are heuristics from holder
snapshots, not proof of a rug or intent.

What Ruggers is for
  - Freeze who held what on FIRST full lookup (Holders + Bundles tags).
  - On later Analyzes, list wallets that sold ≥99% of that first bag.
  - Keep each wallet in its category (creator / similar / multi / funder /
    insider / launch / suspect / single / Flagged RugWatch).
  - Sticky: if they never buy back, they stay in that section permanently
    (this browser). Buy-back → Swing (label kept) → sell again → same section.
  - Upload / Export seller wallets into YOUR RugWatch list (see §12).

How tracking starts (first lookup)
  After a successful full Analyze (Quick OFF) with holders:

  1. Non-LP holders (bag ≥ 0.01% when % known) are stored as baseline.
  2. Bundles tags attach: similar-size, multi-account, insider, shared funder,
     launch-window same-slot, suspect.
  3. Creator wallet (when known) is included.
  4. Each wallet’s first supply % (and balance when known) is frozen.
  5. Origin lane is frozen once (see chart below). Later Analyzes do not
     rename the wallet to another lane (except promote to similar in rare cases).

  Quick mode does not start or update Ruggers (no full holder scan).

Where each Ruggers section comes from (Holders / Bundles)

  | Ruggers section                         | Tied to (at first full Analyze)           |
  |-----------------------------------------|-------------------------------------------|
  | Creator (sold ≥99%)                     | Mint creator (Holders meta / Rugcheck)    |
  | Similar wallets (sellers)               | Bundles similar-size groups               |
  | Multi-account clusters (1 Owner)        | Holders/Bundles owner clusters (multi-ATA)|
  | Shared SOL funder clusters (1-Owner)    | Bundles funding clusters (1-hop SOL)      |
  | Insider-flagged wallets (Rugcheck)      | Rugcheck insider flag (NOT RugWatch)      |
  | Same-slot multi-buys (Launch-window)    | Bundles launch-window / same-slot buys    |
  | Suspect wallets                         | Bundles suspect union (if not more specific)|
  | Single wallets (sellers)                | Plain holders ≥0.01%, none of the above   |
  | Flagged wallets (RugWatch)              | YOUR RugWatch cloud/local + sold ≥99%     |
  | Swing traders                           | Buy-back after ≥99% sell (any origin lane)|

  Priority if a wallet fits more than one lane at first discovery:
    creator → similar → multi → funding → insider → launch → suspect → single

  Flagged (RugWatch) is a separate path (sold while on your RugWatch list),
  not a Bundles category name. Title on the site: Flagged wallets (RugWatch)
  with “RugWatch” in dim yellow.

Does the number of lookups before they sell matter?
  NO. Lookup count does not matter.

  - Sale is always measured against the FIRST full lookup bag only.
  - Dump can appear on the 2nd, 3rd, 10th, or 50th full Analyze — same rule.
  - Intermediate lookups do not reset tracking.

  Example
    Lookup 1: wallet holds 5% (frozen as first bag)
    Lookups 2–9: still holds ~5% → not listed as seller
    Lookup 10: holds 0.02% or gone from the list → SELLER
      (same rule as if they had sold on lookup 2)

What “sold ≥99%” means
  A wallet is listed only if it dumped at least 99% of its FIRST-lookup bag:

    remaining_first_bag ≤ 1% of first bag
    OR the wallet is no longer on the holder list (treated as sold 100%)

  Examples
    First hold 5.00% of supply → still listed with 0.04% → seller (~99.2% sold)
    First hold 5.00% → still listed with 0.10% → NOT listed (only 98% sold)
    First hold 5.00% → gone from top holders → seller (not listed anymore)
    First hold 5.00% → sold out, then later holds 1% again → swing
    First hold 5.00% → still holding until 10th lookup, then sold ≥99%
      → still a seller (lookup number does not matter)

  Wallets that never sold ≥99% are ignored (not shown).

Sell ↔ Swing loop (all sticky origin lanes)
  1. Sell ≥99% of first bag → stay in that section (sticky / permanent if no return)
  2. Buy-back → Swing traders, label KEPT:
       e.g. multi-account · swing · insider · swing · flagged · swing
  3. Sell ≥99% of the swing bag again → back to the SAME origin section
  4. Buy-back again → Swing … (loop continues)

  If they never buy back → they stay in their seller section permanently
  (this browser’s localStorage; until you clear site data).

Labels (examples)
  multi-account · seller   similar · seller   launch-window · seller
  multi-account · swing    flagged · swing    creator · swing
  seller / swing heuristics only — not legal proof of collusion

UI controls (website)
  - No Refresh / Clear mint / Clear all toolbar (removed).
  - Tracked mint: scrollable dropdown (max height; does not fill the screen).
  - Search previous lookup by CA (local only — no network).
  - Left-click mint address (yellow) to copy CA.

Per-section actions (every seller section — not Swing)
  Export (gray)  — download JSON / address list for RugWatch manual import
  Upload (yellow)— CLOUD upload path for that section’s NEW wallets:
                   1) import into YOUR RugWatch local DB
                   2) Push cloud → GitHub wallets_index / cloud shards
                   Requires RugWatch running (default http://127.0.0.1:8790)
                   with GITHUB_TOKEN. Configure rugwatchUrl in config.js.

  After Upload
    - Creator → treated as Flagged (RugWatch) path on this mint
    - Similar / multi / funder / insider / launch / suspect / single
      → stay under their Ruggers category on this mint (also on cloud)
    - On other mints, cloud-listed wallets can enter Flagged (RugWatch)
      when they sell ≥99% (checkbox ON)

Where it lives
  Website only: browser localStorage key adtc_ruggers_track
  (separate from Logs). Not on the ATC server. Not shared across devices.
  Clearing site data resets it.

  Multiple mints: use the scrollable “Tracked mint” dropdown to switch.

Typical workflow
  1. Full Analyze a new mint (Quick off) → baseline freezes (seller sections
     empty until someone dumps).
  2. Later, full Analyze the same mint again (any later run).
  3. Open Ruggers → sellers under Creator / Similar / Multi / Funder /
     Insider / Launch / Suspect / Single / Flagged (RugWatch).
  4. Buy-back → Swing (same label). Dump again → same section.
  5. Upload or Export any seller section → YOUR RugWatch (see §12).

What Ruggers is not
  - Not a live mempool dump alert
  - Not proof wallets are the same person or a “rug team”
  - Not RugWatch itself (RugWatch is the separate wallet DB / website)
  - Not updated by Quick-only searches
  - Not a substitute for Solscan trade history (top-holder snapshots only)
  - Yellow Upload needs YOUR RugWatch instance; not a shared public police DB
  - Insider-flagged (Rugcheck) ≠ Flagged wallets (RugWatch)

12. SAVING SELLER WALLETS TO RUGWATCH (LOCAL + CLOUD)
--------------------------------------------------------------------------------

Subtitle: ATC side only — full RugWatch product guide is separate

Two different tools (do not mix them up)
  Ruggers (this site)
    - Dump spotting for mints you analyze in THIS browser (localStorage).
    - Seller / swing = ≥99% of first bag (see §11). Not a RugWatch score.

  RugWatch (separate app / website)
    - YOUR wallet warehouse (local SQLite + optional GitHub cloud).

      Add/Upload, Monitor, cloud). Do not duplicate that manual here.

Ways to grow YOUR RugWatch list from ATC Ruggers
  A) Yellow Upload on any seller section → RugWatch /api/upload + Push cloud
  B) Export file → RugWatch “Upload manual wallets” → Push cloud
  C) Copy address → RugWatch Add wallet (you set score)

  - Live site: almost always CLOUD only via RUGWATCH_WALLETS_URL
  - Local PC: may merge local DB + cloud → tags [local] / [cloud] / [both]
  - Shows hold % on the mint; does NOT invent a token-level RugWatch rating

What ATC will NOT do
  - Shared public “serial rugger police” DB for all users
  - Host your private RugWatch scores (you curate scores in RugWatch)
  - Sync browser Ruggers lists across devices

Summary
  Ruggers  = temporary dump tracker (this browser).
  RugWatch = your scored wallet list + cloud (see RugWatch docs for scoring).
  ATC flags = match list → Holders / Ruggers Flagged (RugWatch) / Alerts.

13. LOGS
--------------------------------------------------------------------------------

Subtitle: Logs

The Logs tab is a private log of tokens you have already searched.
It sits next to Overview, Holders, Bundles, Alerts, Maps, About, and Ruggers
on the desktop app and the website. Use it to look back at recent analyzes
without typing every mint again.

  Desktop: saved in history_log.json next to the app.
  Website: saved in this browser only (localStorage).

What Logs is for
  - See a list of your last token searches in one place.
  - Review market, holder, bundle, and alert snapshots from those runs.
  - Download the list to a file for your own notes.

When a search is saved
  - A row is added only after a successful full Analyze finishes
    (status shows OK / full data loaded).
  - Failed searches (not found, errors) are not saved.
  - Quick or partial loads wait until the full run completes before
    they appear in Logs.
  - Searching the same token again adds another row (each run is kept
    as its own entry).

What each entry shows
  A snapshot from the moment you searched — not live prices:

    · When you searched (UTC)
    · Symbol, name, mint address, chain
    · DEX / pool when known
    · Price, market cap, liquidity, 24h volume and change
    · Holder concentration and Top1 / Top5 / Top10 %
    · Bundle risk and share %
    · How many top-priority alerts fired
    · Pump.fun status when known
    · Pair link when known

  Plus frozen text “screenshots” saved at lookup time:

    · HOLDERS SNAPSHOT — the Holders report as it looked when you
      Analyzed (top wallets, %, creator, flags when available).
      Long reports may be truncated so 20 entries still fit in storage.
    · BUNDLES SNAPSHOT — the Bundles report as it looked at that same
      moment (totals, risk, suspect signals when available). Also may
      be truncated for space.

  These freeze at search time. They do not update later. To get a fresh
  holders/bundles list, run Analyze again and open Holders / Bundles.

How many entries
  - Logs keeps up to 200 searches.
  - Newest is at the top.
  - When you go past 200, the oldest entry is deleted automatically on each consecutive lookup.

Search previous lookup by CA (website)
  Paste a mint / CA into the Logs search box → Search (or Enter).
  Finds a prior Logs entry saved in this browser only (no network).
  Same idea as Ruggers CA search, separate storage from Ruggers.

Where it lives
  Desktop
  - Saved as history_log.json next to the app.
  Website
  - Saved in your browser only (not on the server, not shared with others).
  - Clearing site data / another browser / private mode starts empty.
  - Different from the small search-bar dropdown (desktop only).

Buttons
  Refresh
    Reloads the Logs list from storage (desktop: history_log.json on
    disk; website: this browser’s saved data). Use if the tab looks
    stale after Analyze, or after you reopened the app. Does not
    re-scan tokens and does not call APIs.

  Clear
    After you confirm, deletes ALL Logs entries for this app/browser.
    Cannot be undone. Desktop empties history_log.json; website clears
    localStorage. Does not affect Overview/Holders for the token you
    currently have open.

  Download
    Saves the current Logs list to a FILE ON YOUR COMPUTER only.
    · .txt — readable text (same layout as the tab, including
      holders/bundles snapshots)
    · .json — full structured data for backup or tools
    Does not delete Logs from the app. Does not re-fetch market data.
    Does NOT upload to RugWatch cloud, GitHub, or any ATC server.
    Private browser/app log only.

Upload vs Logs (important)
  The Logs tab does NOT push wallets to the cloud database.
  Cloud upload (RugWatch GitHub wallet list / index) is on the
  Ruggers tab:

    Ruggers → Creator / Similar / Single → yellow Upload
      → imports into YOUR RugWatch local DB
      → Push cloud → GitHub (cloud wallet count increases)

  Requirements for that Upload: RugWatch website running (default
  http://127.0.0.1:8790), GITHUB_TOKEN set in RugWatch .env, and
  rugwatchUrl in web/config.js pointing at that site.

  Summary
    Logs Download     = save search history to your PC (not cloud)
    Ruggers Upload    = send seller wallets to RugWatch cloud server
                        (GitHub via your RugWatch instance)

How to use it
  1. Analyze a token and wait until the run finishes successfully.
  2. Open the Logs tab — that token should be first on the list, with
     holders and bundles snapshots under the summary.
  3. Keep analyzing other tokens (up to 20 kept).
  4. Use Download if you want a local copy of the log (not cloud).
  5. To grow the RugWatch cloud list, open Ruggers → yellow Upload
     (see §11–§12), not Logs.
  6. Use Clear only if you want an empty Logs list.
  7. Use Refresh if the list did not update after Analyze.

What Logs is not
  - Not a live price feed or live holders feed.
  - Not the place that uploads wallets to RugWatch / cloud (that is
    Ruggers yellow Upload, or RugWatch’s own Push cloud).
  - Not a camera image — “snapshot” means a frozen text copy of the
    Holders / Bundles report at Analyze time.
  - Not a full export of every report field beyond what was stored
    (use Export JSON on desktop for the full current report).
  - Not a safety rating or “safe list.” Past results are for your
    reference only.

15. PREVENTING 429s, BREAKDOWNS, AND SLOW FETCHING
--------------------------------------------------------------------------------

  You cannot eliminate every slowdown, but you can stop most breakdowns and
  429 storms with habits + plan choices. This applies to Actual Token Checker
  (and RugWatch when Upload / flags use cloud).

Why you get 429s and slowness
  | Cause                              | What happens                                      |
  |------------------------------------|---------------------------------------------------|
  | Provider free tiers                 | Too many Helius / Birdeye / DexScreener calls →    |
  |   (Helius, Birdeye, DexScreener)   | HTTP 429 “rate limited”                           |
  | Many full Analyzes at once         | Free host does ~1 heavy job well; parallel clicks |
  |                                    | stack and time out                                |
  | Cold starts (free web sleep)       | First request after idle ~30–90s feels “broken”   |
  | Large RugWatch cloud               | Pull / ATC flags load big JSON → slow or OOM      |
  | Immediate retries after 429        | Makes the limit worse                             |

  429 is usually the API provider refusing more traffic — not your laptop
  and not “the app permanently broken.”

Practical prevention (do these first)
  1. One full Analyze at a time.
     Wait until Holders / Bundles / Alerts finish before the next mint.
     Parallel full runs burn Helius and DexScreener fastest.

  2. Prefer Quick when you only need price / overview.
     Full Analyze = holders + bundles + flags + more provider calls.
     Use full only when you need that depth.

  3. Do not spam Refresh / re-Analyze the same CA.
     Re-running the same mint in a loop is a common 429 trigger.

  4. Keep RugWatch cloud curated, not millions of wallets.
     Ideal: hundreds – low tens of thousands.
     Heavy: 100k–200k+ (slower Push/Pull, slower ATC flags).
     Upload from Ruggers only when you care about those sellers.

  5. Wake services before heavy work.
     Open ATC (and RugWatch if you will Upload) once after idle, wait for
     cold start, then Analyze / Upload. Avoid Upload while RugWatch is
     still sleeping.

  6. Site passcode on (WEB_API_TOKEN).
     Cuts random traffic that burns your keys.

  7. Do not share free-tier API URLs publicly.
     Bots + many people clicking Analyze share the same rate limit.

Keys / tiers (biggest real fix for 429s)
  | Provider      | Free reality                         | Better for stability              |
  |---------------|--------------------------------------|-----------------------------------|
  | Helius        | Monthly caps + RPC rate limits       | Paid / higher tier if you Analyze |
  |               |                                      | a lot (best ROI for 429s)         |
  | Birdeye       | Tight rate limits                    | Optional; skip if not needed      |
  | DexScreener   | Public, easy to 429                  | Full mint + wait; do not hammer   |
  | GitHub cloud  | Fine for light Push/Pull             | PAT set; do not Push every few    |
  |               |                                      | seconds                           |
  | Render free   | Sleep + small RAM                    | Paid always-on if you hate cold   |
  |               |                                      | starts                            |

  Rule of thumb: if you hit 429s often, upgrade Helius before anything else.
  Holders / deep work live there.

Hosting habits (e.g. free Render)
  - Treat free as personal, light use — not multi-user production.
  - Expect sleep after ~15 minutes idle.
  - Avoid two free services both cold at once (ATC + RugWatch): open
    RugWatch first, then ATC Upload.
  - Large cloud Pull on free can OOM / restart — pull when idle, not mid-
    Analyze.

Stable day-to-day workflow
  1. Open ATC + RugWatch (wake cold starts).
  2. One mint → Quick or full Analyze.
  3. Wait for finish.
  4. If Ruggers has new sellers → Upload once (do not spam).
  5. Pause between mints (even 5–15 seconds helps public APIs).
  6. Do not Pull cloud every few minutes.

What “slow” usually is (not always a crash)
  | Symptom                         | Likely cause                          |
  |---------------------------------|---------------------------------------|
  | 30–90s first load               | Free cold start                       |
  | Holders empty / provider notes  | 429 or missing / exhausted API key    |
  | Upload hangs                    | RugWatch asleep                       |
  | Flags slow                      | Large cloud index / many shards       |
  | Random timeouts                 | Concurrent Analyzes or free CPU limit |

Optional upgrades if you want fewer headaches
  1. Helius paid (best ROI for 429s).
  2. Always-on host for ATC (and RugWatch if you Upload daily).
  3. Use Logs instead of re-fetching the same mint.
  4. (Optional code later) retry with backoff on 429, single-flight Analyze
     queue, short TTL cache per mint.

Short checklist
  [ ] One Analyze at a time
  [ ] Do not re-check the same CA every few seconds
  [ ] Wake RugWatch before Upload
  [ ] Passcode on
  [ ] Cloud not huge unless you need it
  [ ] Helius key valid and not over monthly cap
  [ ] Accept free-tier cold starts or pay for always-on

Bottom line
  Breakdowns and 429s are mostly rate limits + free hosting + concurrent
  full scans — not Logs size or UI colors. Steady one-at-a-time use +
  Helius headroom + a smaller cloud prevents most of it.

16. COMMON QUESTIONS
--------------------------------------------------------------------------------

Q: Can visitors add their own Helius keys?
A: No. Keys stay on the server. Gear = site passcode only (if enabled).

Q: Why DexScreener 429 / bad response?
A: Public rate limits. Wait, use Quick, use full mint, enable site passcode
   so fewer people spam Analyze. Cache reduces repeat hits.
   See §15 for a full guide (429s, cold starts, cloud size, workflow).

Q: Why is Analyze slow or timing out?
A: Free host cold start, concurrent full Analyzes, provider 429s, or a large
   RugWatch cloud. Follow §15 (one Analyze at a time, wake services, Helius
   headroom).

Q: Do view counts reset daily?
A: Total views do not reset daily. “Unique today” does. Free hosting may
   reset totals after redeploy.

Q: Why is LP unlocked on a Pump.fun coin?
A: LP-lock check is classic pool lock %, not a full bonding-curve model.
   On desktop Alerts, that LP-unlock warning is skipped for Pump.fun /
   PumpSwap pools (and classic “…pump” mints) because those venues do
   not map cleanly to locked LP pools.

Q: Does a clean Alerts tab mean safe?
A: No. Absence of flags ≠ safety.

Q: Where did my Logs go after reinstall or moving the app?
A: Desktop Logs are saved in history_log.json next to the app. Website
   Logs live in the browser only. If you move the app folder or clear
   site data, the list may be gone. Use Download to keep a backup.

Q: Is there a serial-rugger database? Who creates it?
A: Yes — RugWatch on your desktop is that database (local SQLite, wallets
   you flag). The website Ruggers tab does not create or update it. Copy
   seller addresses from Ruggers and add them in RugWatch yourself (§12).
   The live website host does not keep a shared serial-rugger DB for all
   users.

Q: If a wallet sells on my 10th lookup, do they still count as a seller?
A: Yes. Lookup count does not matter. Sales are always measured against
   the first full Analyze bag (see §11).
